## 1. Schema (Drizzle TypeScript)

```typescript
import {
  pgTable,
  pgEnum,
  uuid,
  varchar,
  boolean,
  timestamp,
  text,
} from "drizzle-orm/pg-core";

// --- Enums ---

export const roleEnum = pgEnum("role", ["TA", "FA", "CA", "CF", "AP"]);

export const authTokenTypeEnum = pgEnum("auth_token_type", [
  "magic_link",
  "password_reset",
]);

// --- Users ---
// Auth-relevant fields only per spec.
// NOTE: password_hash is added here — see Ambiguity Q1.

export const usersTable = pgTable("users", {
  id:            uuid("id").defaultRandom().primaryKey(),
  email:         varchar("email", { length: 255 }).unique().notNull(),
  role:          roleEnum("role").notNull(),
  modules:       text("modules").array().notNull().default([]),
  hasPassword:   boolean("has_password").notNull().default(false),
  passwordHash:  varchar("password_hash", { length: 255 }),        // nullable; see Q1
  isActive:      boolean("is_active").notNull().default(true),
  keepLoggedIn:  boolean("keep_logged_in").notNull().default(false),
  lastLoginAt:   timestamp("last_login_at"),
  createdAt:     timestamp("created_at").notNull().defaultNow(),
  updatedAt:     timestamp("updated_at").notNull().defaultNow(),
});

// --- Auth Tokens ---
// Unified table for magic_link and password_reset (D-AUTH-001).
// Token stored as bcrypt hash of the raw token sent in the email.
// Single-use enforced via used_at; expiry enforced via expires_at (D-ARCH-047).

export const authTokensTable = pgTable("auth_tokens", {
  id:         uuid("id").defaultRandom().primaryKey(),
  userId:     uuid("user_id").notNull().references(() => usersTable.id),
  token:      varchar("token", { length: 512 }).unique().notNull(),  // hashed
  type:       authTokenTypeEnum("type").notNull(),
  expiresAt:  timestamp("expires_at").notNull(),                     // +30 min from created_at
  usedAt:     timestamp("used_at"),                                  // NULL = unused
  createdAt:  timestamp("created_at").notNull().defaultNow(),
});

// --- Sessions ---
// Cookie-based, server-side session store (D-ARCH-019 — NOT JWT).
// Session ID is a random UUID stored in an httpOnly + secure + sameSite=strict cookie.
// Role-based expiry set at creation (D-ARCH-017); keep_logged_in respected at creation (D-ARCH-018).
// See Ambiguity Q3 for sliding vs. fixed expiry question.

export const sessionsTable = pgTable("sessions", {
  id:           uuid("id").defaultRandom().primaryKey(),             // value stored in cookie
  userId:       uuid("user_id").notNull().references(() => usersTable.id),
  expiresAt:    timestamp("expires_at").notNull(),                   // role-based: 12h or 48h
  createdAt:    timestamp("created_at").notNull().defaultNow(),
  lastActiveAt: timestamp("last_active_at").notNull().defaultNow(),  // updated on each request; see Q3
});
```

## 3. Component Plan

|-----------|-------------|
| `StaffLoginPage` | Email input field + "Send magic link" CTA. Secondary link: "Sign in with password" (only shown if `has_password = true`, per D-AUTH-004 / D-ARCH-058). |

|-----------|-------------|
| `ApplicantLoginPage` | Separate entry point at `/apply` — email input + "Send magic link". No password option (AP is magic link only, always). |

## 4. Ambiguities + Questions

### Q1 — STOP: `password_hash` not in user entity spec

**Type:** Schema — STOP and ask.

The `users` entity in the spec lists `has_password BOOLEAN` but does not include `password_hash`. If password is an optional secondary auth method, the hash must be stored somewhere. I have added `password_hash VARCHAR(255) NULL` to `usersTable` as the most natural location, but this is not confirmed in the spec.

**Options:**
- (A) Confirm `password_hash` lives on `users` (my proposal — simplest)
- (B) Store password hashes in a separate `user_passwords` table (adds a join but keeps `users` clean)

**Need:** Explicit confirmation of (A) or (B) before schema is finalized.

---

### Q2 — STOP: `modules[]` column type

**Type:** Schema — STOP and ask.

The spec lists `modules[] ARRAY` for module access control but does not define what the values are. I have used `text("modules").array()` (array of strings), but I need to know:

- What are the valid module identifiers? (e.g., `"CIS_FUND_SETUP"`, `"SCHOLARSHIPS_APPLICATIONS"`, etc.)
- Should this be an array of strings, or a PostgreSQL ENUM array?
- Is an empty array the correct default for a newly created user, or should it default to role-based module access?

**Need:** Module identifier list + confirmation of column type before schema is finalized.

---

### Q3 — STOP: `keep_logged_in` behavior — fixed vs. sliding expiry

**Type:** Workflow — STOP and ask.

D-ARCH-018 says `keep_logged_in` "extends session to role-based timeout limit." This is ambiguous between two behaviors:

- **(A) Fixed expiry:** Session is created with a longer TTL (e.g., 48h for TA instead of 12h). `lastActiveAt` is updated on each request but has no effect on `expiresAt`. Session dies at the fixed time regardless of activity.
- **(B) Sliding expiry:** `expiresAt` is reset on each request to `now() + timeout`. Session stays alive indefinitely while the user is active.

I have added `last_active_at` to `sessionsTable` to support either model, but the session renewal logic differs significantly between them.

**Need:** Confirmation of (A) or (B) before writing session middleware.

---

### Q4 — Clarification: AP login entry point isolation

**Type:** UI/workflow — flagging, not blocking.

The spec mentions the external questionnaire as a separate public portal. I have routed AP auth to `/apply` and `/apply/sent` to keep it isolated from staff auth at `/login`. This is my decision — flagging for confirmation.

If the AP portal is a fully separate domain or subdomain, the routing assumptions above change. Is `/apply` on the same domain as staff, or is it a separate deployment?

### Q5 — Clarification: TA/FA-initiated password reset flow

**Type:** Workflow — flagging, not blocking.

D-AUTH-003 lists "TA/FA-initiated" as the third password reset entry point. D-AUTH-005 confirms FA scope. My assumption:

- TA/FA triggers a `password_reset` token for the target user from the user management screen
- System sends the reset email to the target user's address (not to TA/FA)
- Flow is otherwise identical to the forgot-password flow from the target user's perspective

If the TA/FA flow instead displays the reset link on-screen (like the magic link copy-to-clipboard fallback), the component plan needs an additional admin-facing display component.

**Need:** Confirm whether TA/FA-initiated reset sends email to user or displays link to admin.
