## Section 1 — Corrected Relay Model

The relay model in ark0183 replit.md described John copying files between
Chat and Agent. That model is corrected here. The real model:

### Agent → Chat

Agent writes its report or questions directly to `/claude-comms/` and
commits to GitHub. John clicks "Push" in Replit's git widget. John gives
Chat the raw GitHub URL. Chat reads it. John does not read it.

Agent never waits for a direct human response. All responses come via
the `/ark/` channel (see below).

### Chat → Agent

Chat writes an instruction or response file to the Ark (ark1###.*).
John does a single `git push` from his local clone. Agent reads the
new file from `/ark/` on its next action.

### John's role

John is the trigger — not the message carrier. His two actions:
- **Replit Push** = "I authorize Agent's output to be released for Chat review"
- **git push** = "I authorize Chat's instruction to be released to Agent"

John does not read Agent's raw output. John does not answer Agent's
questions directly. All question routing goes through Chat.

### Update replit.md

Update the "Key Locked Decisions" section of replit.md to replace the
old file exchange entry with:

> **Relay model:** Agent writes all responses and reports to `/claude-comms/`
> and commits to GitHub. John pushes to release. Chat reads via raw GitHub URL.
> Chat writes instruction/response files (ark1###) to Ark. John git-pushes to
> release. Agent reads from `/ark/`. John is the trigger, not the message carrier.
> Agent never expects a direct human response — all answers arrive via `/ark/`.

## Section 2 — Plan-Gate Protocol

Before building anything in any sub-phase, Agent must:

1. Read the GWT scenario file for the sub-phase from `/ark/`
2. Read BUILD-STATUS.md from `/ark/`
3. Enter plan mode — produce a plan with no code written
4. Write the plan to `/claude-comms/ark[N].plan.md` with this structure:

```
# Plan — [Sub-phase ID] [Sub-phase label]

## My Understanding
[Restate what is being built in plain language]

## Schema Changes
[List all Drizzle schema additions/changes]

## Routes
[Page routes and API routes]

## Components
[List of components to build]

## Build Sequence
[Ordered steps — what gets built in what order]

## Questions
[Any ambiguities — or "None" if clean]

## Confirmation
I have read the GWT scenario file and BUILD-STATUS.md.
I have not written any feature code.
I am waiting for approval before proceeding.
```

5. Commit to GitHub, push
6. Wait — do not build until an approval file arrives in `/ark/`

If Agent has questions in the plan file, Chat reviews them and either
answers from spec or routes to John. Agent does not expect John to answer
directly. The answer arrives as an ark1### file in `/ark/`.

When approval arrives, Agent proceeds to build, then writes a development
report to `/claude-comms/` on completion.

## Section 4 — Chat-as-Filter Model

This is a correction to the implied model in ark0183.

**Old model (incorrect):** Agent surfaces questions → John answers
**Correct model:** Agent surfaces questions → Chat reads → Chat answers
from spec if possible → only unresolvable questions reach John,
already reframed into a yes/no or simple call

**What this means for Agent:**

When you have questions, write them clearly in the plan file or report.
Do not expect a human to answer directly in a chat window. An answer
will arrive as an ark1### file in `/ark/`. The response may come quickly
(Chat resolved it from spec) or may take longer (Chat needed John's call).
Either way, the answer comes through the file channel — not verbally.

Do not build past a question. Write the plan with questions included,
commit, and wait for the approval file.
